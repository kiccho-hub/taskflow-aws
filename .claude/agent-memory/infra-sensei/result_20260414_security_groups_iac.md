## ✅ 得意だったこと（Keep）

- **セキュリティグループの依存関係を正確に構築**
  Frontend と Backend を分離し、Backend から必要なポート（API 8000/TCP）への接続を厳密に制限。アーキテクチャの意図が正確に HCL に反映されている。

- **Terraform 構文の正確な記述**
  `aws_security_group_rule` リソースでの細粒度制御と `aws_security_group` での一括定義の使い分けが上手。Task 1 から継続した品質レベルを維持している。

- **RDS・ElastiCache を先読みした設計**
  現時点では未実装の RDS・ElastiCache 用セキュリティグループをあらかじめ設計に組み込み、将来の構成変更に耐える柔軟性を確保。

- **タグのベストプラクティス（継続習慣化）**
  `locals { common_tags }` による共通化が Task 1 から定着し、Task 2 でも自然に適用されている。リソース管理の品質が一貫している。

## 🔧 改善ポイント（Try）

- **セキュリティグループの検証を詳細に記録する習慣**
  実装後、`terraform apply` で実際に作成されたルールを確認し、AWSコンソール側と照合する確認作業をドキュメント化するとさらに信頼度が高まる。

- **Ingress ルール複数化時の保守性向上**
  複数ソースを許可する場合、リスト化して保守性を上げる工夫。コメントで「なぜこのポートか」という意思が残ると、後で他者がコードを読むときに理由が明確になる。

## 📚 今日の学習ポイント

- セキュリティグループはステートレスファイアウォール。Ingress（受け入れ）と Egress（送出）の制御を分離。

- アーキテクチャ段階で「FE→ALB」「ALB→Backend」「Backend→RDS」という通信パターンを先に整理してから SG を引く設計思想を習得。

- `source_security_group_id` を使ってセキュリティグループ間通信を参照する際の依存関係を理解。

- Task 1 コンソール版との差分を自力で把握し、IaC 版の改善点（冗長性削除、タグの共通化）を認識。

## ⚠️ 要注意ポイント（苦手分野）

- **ALB と複数セキュリティグループの分散配置**
  Task 7（ALB）で複数のターゲットグループを扱うとき、SG 分離のパターンを再度検討する必要あり。

- **Egress（送出）ルールの細粒度設計**
  デフォルトでは「すべてのトラフィック許可」だが、本番環境では送出先を制限する設計考慮が必要。今回の dev 環境と異なる本番設計パターンを後で習得すること。
