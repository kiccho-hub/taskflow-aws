✅ 得意だったこと（Keep）
- GitHub Actions + AWS OIDCの仕組みを「パスポート」の例え話で理解し、本質を掴んだ
- 3つのエラー（AssumeRoleWithWebIdentity / ServiceNotFoundException / NoSuchBucket）を粘り強く自力でデバッグして完全解決した
- リポジトリ名・バケット名のズレという「細かい不一致」を自分で発見できた（これは熟練エンジニアも詰まるポイント）
- アクセスキーではなくOIDCを使う理由（一時認証トークン・シークレット管理不要）を本質から理解した

🔧 改善ポイント（Try）
- IAMロールの信頼ポリシー設定時、リポジトリ名などのパラメータは事前にメモして貼り付けるとケアレスミスが減る
- 複数の設定値（バケット名・クラスター名・サービス名）をGitHub Secretsに登録する前に、AWSコンソールで正確な名前を確認するチェックリストを作ると効率的

📚 今日の学習ポイント
- OIDC認証：短命トークンを使うことでシークレット漏洩リスクをゼロにできる
- IDプロバイダー：「GitHubはこの人を信頼する」という証明書をAWSに登録する仕組み
- エラーログの読み方：エラーメッセージには必ず「何が」「どこで」失敗したかが書かれている

⚠️ 要注意ポイント（苦手分野）
- 設定値の名前ズレ（今回は aws-demo → taskflow-aws、バケット名の末尾 `-an`）：複数のサービスをまたいで設定するときに発生しやすい
