## 学習を通じて習得した技術

### ネットワーク層
- VPC設計（CIDR、パブリック/プライベートサブネット）
- セキュリティグループの連鎖設計（ALB→ECS→RDS→Redis）
- NAT Gateway・IGW・ルートテーブル

### コンピューティング層
- ECS Fargate（タスク定義、サービス、desired count管理）
- ECRコンテナレジストリ管理
- ALBパスベースルーティング（/api/* → backend, /* → frontend）

### データ層
- RDS PostgreSQL 16（Multi-AZ、暗号化、バックアップ）
- ElastiCache Valkey（クラスターモード、ノードタイプ選択）

### アプリケーション層
- Cognito ユーザープール（3ロール: Guest/User/Admin、新UI対応）
- S3 + CloudFront（静的サイトホスティング、OAC設定）

### 自動化・運用層
- GitHub Actions + AWS OIDC（短命トークン、アクセスキー不要）
- CloudWatch（ダッシュボード、アラーム、SNS通知）

## 得意分野（継続して伸ばすべき強み）

1. **デバッグ力** — 本番エラーに直面しても諦めずに原因を特定できる（Task7: 4エラー、Task11: 3エラーの自力解決）
2. **設計理解** — 「なぜそうするのか」を本質から理解しようとする姿勢（S3+CloudFront設計、OIDC仕組み）
3. **UI変更への適応** — AWS コンソールの新UI変更を自力で発見・対応できる（Cognito, ElastiCache）

## 継続的に改善が必要な点

1. **作成後の確認チェックリスト** — 重要設定（タグ、URI、設定値）を作成後に必ず確認する習慣
2. **複数リソース間の設定名一致** — 複数サービスにまたがる設定値の名前ズレに注意
3. **タスク間依存関係の事前確認** — 前のタスクで必要なリソースが揃っているか確認してから着手

## 先生からのコメント

全12タスクのコンソールフェーズを完走しました！これはAWSインフラの「ほぼ全層」を手で触った経験です。
VPC（土台）→セキュリティグループ（壁）→RDS・Redis（データ）→ECS（アプリ）→ALB（玄関）→Cognito（鍵）→S3+CloudFront（配達）→CI/CD（自動化）→CloudWatch（監視カメラ）という、
本物のシステムと同じ構造をゼロから作り上げました。

次のIaCフェーズでは、この手で作った経験が「コードを読んだときにリソースが頭に浮かぶ」という最高の武器になります。

## 次のステップ（IaCフェーズへ）

1. **Task1から順にTerraformコードを書く**
2. `terraform init` → `terraform plan` → `terraform apply` の流れを習得
3. コンソールで作ったリソースをTerraformで再現し、`terraform import` も体験する
4. `locals { common_tags }` + `merge()` パターンでタグを一元管理する
