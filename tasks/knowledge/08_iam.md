# Knowledge 08: IAMとアクセス制御

Task 8（ECSサービス・タスク定義）の前に理解しておくべきIAMの概念。

---

## IAMとは

**Identity and Access Management** — AWSリソースへのアクセス権限を管理するサービス。「誰が」「何を」「どのAWSリソースに対して」できるかを定義する。

---

## IAMの主要概念

| 概念 | 説明 |
|------|------|
| ユーザー | 人間が使う長期的な認証情報 |
| グループ | ユーザーをまとめてポリシーを一括付与 |
| ロール | サービスや一時的な操作のための権限（パスワードなし） |
| ポリシー | 何を許可/拒否するかを定義したJSONのルール |

---

## ユーザーとロールの使い分け

**IAMユーザー：**
- 人間が長期間使う（AWS CLIの設定や開発者のコンソールアクセス）
- アクセスキー（長期有効な秘密情報）を持つため、漏洩リスクが高い

**IAMロール：**
- AWSサービス（ECS・Lambda・EC2など）が使う
- 一時的な認証情報（STS）を使うため、漏洩しても短時間で無効になる
- アクセスキーが存在しない

原則：「人間にはユーザー（またはSSO）、AWSサービスにはロール」。アプリが動くコンテナにアクセスキーを埋め込むのは最悪のアンチパターン。

---

## ポリシーの種類

**管理ポリシー（Managed Policy）：**
AWSが作成・管理する定義済みポリシー。`arn:aws:iam::aws:policy/...` の形式で参照する。
- `AmazonECSTaskExecutionRolePolicy`: ECSタスク実行に必要な最小限の権限（ECR pull, CloudWatch Logs書き込み）

**インラインポリシー：**
特定のロールに直接埋め込むポリシー。他のロールと共有しない独自の権限に使う。

---

## 最小権限の原則

必要最低限の権限だけ付与するセキュリティの基本原則。

**やってはいけないこと：**
```json
{ "Effect": "Allow", "Action": "*", "Resource": "*" }
```
「全てのAWSリソースへの全操作を許可」= セキュリティホール。コンテナが侵害された際にAWS全体が危険にさらされる。

**あるべき姿：**
```json
{
  "Effect": "Allow",
  "Action": ["s3:GetObject", "s3:PutObject"],
  "Resource": "arn:aws:s3:::my-bucket/*"
}
```
特定のバケット内のオブジェクトの読み書きだけ許可。

---

## AssumeRole（信頼ポリシー）

ロールを使うには「誰がこのロールを引き受けられるか（AssumeRole）」を信頼ポリシーで定義する。

ECSタスクのロールの場合：
```json
{
  "Principal": { "Service": "ecs-tasks.amazonaws.com" },
  "Action": "sts:AssumeRole"
}
```
「ECSタスクサービスがこのロールを引き受けられる」という意味。これがないと、いくらポリシーを付けてもECSはそのロールを使えない。

---

## ECSの2種類のロールの使い分け（復習）

Task 8で必要になるため詳しく整理しておく。

**タスク実行ロール（execution_role_arn）：**
ECS**コントロールプレーン**（AWSのECS基盤）が使う権限。タスク起動前の準備作業に使う。

具体的に何をするか：
1. ECRからDockerイメージをpull
2. Secrets ManagerやSSMからシークレットを取得してコンテナに渡す
3. CloudWatch Logsにロググループを作成し、ログを書き込む

`AmazonECSTaskExecutionRolePolicy`をアタッチすれば上記は全てカバーされる。

**タスクロール（task_role_arn）：**
起動した**コンテナ内のアプリケーション**が使う権限。アプリのコードから`AWS SDK`でAWSサービスを呼ぶときに使われる。

具体的な例：
- バックエンドが画像をS3にアップロードする → S3へのPutObject権限が必要
- バックエンドがSQSキューにメッセージを送る → SQS SendMessage権限が必要

アプリがAWSサービスを一切呼ばない場合は不要。

---

## IAM設計の実務チェックリスト

- [ ] アクセスキーをコードにハードコードしていない
- [ ] 本番用ロールの権限が最小限になっている
- [ ] 「何かよく分からないからAdministratorAccess付けとこう」はNG
- [ ] ロールに付けるポリシーのResourceは具体的なARNで絞っている
- [ ] タスク実行ロールとタスクロールを混同していない
