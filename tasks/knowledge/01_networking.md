# Knowledge 01: AWSネットワーク基礎

Task 1（VPC・サブネット・ゲートウェイ）の前に理解しておくべき概念。

---

## VPC（Virtual Private Cloud）

AWSアカウント内に作る「自分専用のプライベートネットワーク空間」。デフォルトで外部から遮断されており、自分でルールを設計する。

**CIDRブロック**とは、VPC内で使えるIPアドレスの範囲を表す表記。
`10.0.0.0/16` の場合：
- `10.0.0.0` が基点のIPアドレス
- `/16` がネットワーク部のビット数（残り16ビット = 2^16 = 65,536個のIP）

よく使うサイズ目安：

| CIDR | IPの数 | 用途 |
|------|--------|------|
| /16 | 65,536 | VPC全体（大きめに取る） |
| /24 | 256 | サブネット単位（1AZに1つ程度） |
| /28 | 16 | 最小構成（学習用） |

VPCは一度作ると**CIDRを後から変更できない**。最初に/16程度の広めの範囲を取っておくのがベストプラクティス。

## サブネット

VPC内をさらに分割した小さなネットワーク区画。リソース（EC2・RDS・ECSなど）はサブネットに配置する。

**パブリックサブネット**と**プライベートサブネット**の違いは「インターネットゲートウェイへのルートがあるか否か」だけ。

| | パブリックサブネット | プライベートサブネット |
|--|---------------------|----------------------|
| インターネットへの直接通信 | できる | できない |
| 外からの直接アクセス | できる（IGW経由） | できない |
| 配置するリソース | ALB、NAT Gateway、踏み台EC2 | ECS、RDS、ElastiCache |

### なぜ2つのAZに分散させるのか

**アベイラビリティゾーン（AZ）** = AWSのデータセンター群の単位。東京リージョン(`ap-northeast-1`)には1a・1c・1dの3AZがある。

1つのAZが物理的な障害（電源・冷却・ネットワーク機器の故障）を起こしても、別AZのリソースは影響を受けない。これが**高可用性（High Availability）**の基本。

ALB・RDS Multi-AZ・ECSサービスは「2AZ以上にまたがることが必須」または「推奨」されているため、最初からサブネットを2AZに作っておく。

---

## インターネットゲートウェイ（IGW）

VPC ↔ インターネット間の「出入口」となるコンポーネント。VPCに1つだけアタッチする。

IGW自体は単なる出入口なので、これだけでは通信できない。**ルートテーブルでIGWへの経路を設定したサブネット**だけがインターネットと通信できる。

---

## NAT Gateway

プライベートサブネットのリソースがインターネットに**アウトバウンドだけ**できる仕組み。

```
プライベートサブネット → NAT → IGW → インターネット（外向きのみ）
インターネット → プライベートサブネット（直接入れない）
```

**なぜ必要か：** ECSコンテナがDockerHubからイメージをpullしたり、外部APIを呼んだりするには外向きの通信が必要。でも外からECSに直接アクセスされたくない。NATはその「一方通行」を実現する。

**コスト注意：** NAT GatewayはAWSの中で**費用がかかりやすいリソースの1つ**。
- 存在するだけで約$33/月（固定費）
- データ転送量に応じた従量課金も発生
- 学習環境では、使わない時間は削除するかNATなしの構成にするとコスト節約になる。

---

## ルートテーブル

「このIPアドレス宛の通信はどこに転送するか」を定義する経路表。サブネットに1つ関連付ける。

**パブリックサブネットのルートテーブル：**
```
送信先: 0.0.0.0/0（全ての宛先）→ ターゲット: igw-xxxxxxxx（IGW）
送信先: 10.0.0.0/16（VPC内） → ターゲット: local（VPC内ルーティング）
```

**プライベートサブネットのルートテーブル：**
```
送信先: 0.0.0.0/0 → ターゲット: nat-xxxxxxxx（NAT Gateway）
送信先: 10.0.0.0/16 → ターゲット: local
```

`local` ルートはデフォルトで存在し、同一VPC内の通信を処理する。

## TaskFlow でのネットワーク構成

```
インターネット
    │
[Internet Gateway]
    │
VPC: 10.0.0.0/16
    ├── Public Subnet AZ-a (10.0.1.0/24)  ← ALB, NAT Gateway
    ├── Public Subnet AZ-c (10.0.2.0/24)  ← ALB
    ├── Private Subnet AZ-a (10.0.10.0/24) ← ECS, RDS primary, Redis
    └── Private Subnet AZ-c (10.0.11.0/24) ← ECS, RDS standby
```

**CIDR範囲の設計思想：**
- パブリックは `10.0.1.x` / `10.0.2.x`（1〜9番台）
- プライベートは `10.0.10.x` / `10.0.11.x`（10番台）
- 見ただけでパブリック/プライベートが分かるようにする慣習

## よくある疑問

**Q. VPCはいくつ作れる？**
デフォルトでリージョンあたり5個まで。サービスごとに別VPCを作る設計もあるが、最初は1VPCに集約するシンプルな構成が分かりやすい。

**Q. サブネットとAZは1対1か？**
1つのサブネットは必ず1つのAZに属する。1つのAZに複数のサブネットを作ることは可能（パブリック・プライベートをAZごとに分ける場合など）。

**Q. パブリックIPとElastic IPの違いは？**
パブリックIPはインスタンス起動のたびに変わる。Elastic IPは固定の公開IPアドレスで、NAT GatewayやEC2に割り当てられる（割り当てていないと課金される）。
