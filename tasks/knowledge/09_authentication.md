# Knowledge 09: 認証とCognito

Task 9（Cognito認証設定）の前に理解しておくべき概念。

---

## 認証vs認可

| 概念 | 英語 | 意味 |
|------|------|------|
| 認証（Authentication） | AuthN | 「あなたは誰ですか？」を確認 |
| 認可（Authorization） | AuthZ | 「あなたには何の権限がありますか？」を確認 |

ログイン処理は認証。ログイン後に「管理者だけ削除できる」はアプリ側の認可ロジック。Cognitoは認証と一部の認可（グループ情報の提供）を担う。

---

## なぜ自前で認証を作らないか

認証実装には多くの落とし穴がある：
- パスワードのハッシュ化（bcrypt, argon2の適切な使用）
- ブルートフォース攻撃への対策
- セッションの安全な管理
- トークンの署名・検証
- パスワードリセットフローのセキュリティ
- MFAの実装

これらを全て自前で正しく実装するのは難易度が高く、ミスがセキュリティ事故に直結する。Cognitoのようなマネージドサービスに任せる方が安全。

---

## JWTとは

**JSON Web Token** — ユーザー情報を含む署名付きのトークン。

構造：`ヘッダー.ペイロード.署名`（Base64エンコード）

ペイロードの例：
```json
{
  "sub": "user-uuid-xxxx",
  "email": "user@example.com",
  "cognito:groups": ["User"],
  "exp": 1234567890
}
```

**署名の仕組み：** Cognitoの秘密鍵で署名されているため、改ざんされると検証で弾かれる。バックエンドはCognitoの公開鍵で署名を検証するだけで良い。セッションをDBに保存しなくてもユーザー情報が分かる。

**有効期限：** アクセストークン（1時間）・IDトークン（1時間）・リフレッシュトークン（30日）。有効期限が短いので漏洩しても被害を限定できる。

## Cognitoの構成要素

**ユーザープール（User Pool）：**
ユーザーの認証情報（メール・パスワード）を管理するDB。ログイン・サインアップ・パスワードリセットなどの機能を提供する。

**アプリクライアント（App Client）：**
特定のアプリ（例: React フロントエンド）がユーザープールにアクセスするための識別子。クライアントごとに認証フローや許可するスコープを制御できる。

SPAからの認証の場合、クライアントシークレットを生成**しない**。理由：ブラウザで動くJavaScriptはシークレットを安全に保持できない（ソースコードが全て見えるため）。シークレットが必要なのはバックエンドからCognitoを呼ぶ場合。

**ユーザーグループ：**
ユーザーをグループに所属させ、そのグループ情報をトークンに含める。アプリ側の認可ロジックに使う。

---

## Cognitoの認証フロー（SPA）

```
1. ユーザーがメール/パスワードを入力
2. React が Cognito の /oauth2/token エンドポイントに送信
3. 認証成功 → IDトークン・アクセストークン・リフレッシュトークンを受け取る
4. React はリクエストごとに Authorization: Bearer <IDトークン> をヘッダーに付ける
5. バックエンドが Cognito の公開鍵でトークンを検証
6. グループ情報から権限チェック
```

## パスワードポリシーの設定基準

| 設定 | 開発 | 本番推奨 |
|------|------|--------|
| 最小文字数 | 8 | 8以上 |
| 大文字必須 | 任意 | 推奨 |
| 特殊文字必須 | 任意 | 推奨 |
| MFA | 不要 | 任意（管理者は必須推奨） |

過度に複雑なパスワード要件はユーザー体験を悪化させる。NIST SP800-63Bでは「長さ重視・複雑さ強制しすぎない」が推奨されている。

## ホストされた認証UIとカスタムUI

**ホストされた認証UI：**
Cognitoがホストするログインページをiframeや別タブで開く。実装が最も簡単だがデザインのカスタマイズが限定的。

**Amplify UIライブラリやカスタム実装：**
Amplify SDKや`amazon-cognito-identity-js`を使ってReactコンポーネントとして認証UIを実装する。デザイン自由度が高いが実装コストがかかる。

TaskFlowでは独自UIを使う前提で「ホストされた認証ページ」は設定しない。

---

## ユーザープールIDとクライアントIDの違い

| 項目 | 形式 | 用途 |
|------|------|------|
| ユーザープールID | `ap-northeast-1_XxxxxxxxxX` | バックエンドのトークン検証に使う（JWKSエンドポイントの構成に必要） |
| クライアントID | 英数字26文字程度 | フロントエンドのSDK設定に使う |

両方ともアプリの設定に必要な値。コンソールから確認してメモしておく。
