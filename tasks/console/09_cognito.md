## ハンズオン手順

### Step 1: ユーザープールの作成

1. AWSコンソール → **「Cognito」** → **「ユーザープールを作成」**

---

#### Step 1-1: アプリケーションを定義

新UIでは最初にアプリケーションタイプを選択します。これがウィザード全体の設定（クライアントシークレットの有無・認証フローなど）を自動的に決定する重要な選択です。

**アプリケーションタイプ：**

| 選択肢 | 説明 | クライアントシークレット |
|--------|------|----------------------|
| **Single-page application** | React/Vue などブラウザで動くSPA | なし（Public Client） |
| Traditional web application | サーバーサイドで動くWebアプリ | あり（Confidential Client） |
| Machine to Machine | APIやバッチ処理など非対話型 | あり（Client Credentials） |
| Android | Androidネイティブアプリ | なし |
| iOS | iOSネイティブアプリ | なし |

TaskFlowのフロントエンドはReact SPAです。**「Single-page application」を選択してください。**

> **なぜSPAを選ぶのか：** ブラウザで動くReactアプリはクライアントシークレット（パスワードのようなもの）を安全に保持できません。JavaScriptコードはブラウザの開発者ツールで誰でも見られるからです。SPAを選ぶと「クライアントシークレットなし」のPublic Clientとして設定され、PKCE（ピクシー）という安全な認証フローが使われます。

> **🚨 重要：** このアプリケーションタイプの選択によって後続の設定のデフォルト値が自動的に変わります。間違えると認証方式が変わるので注意してください。

**アプリケーション名：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| アプリケーション名 | `taskflow-web-client` | アプリクライアントの名前。後から変更できる |

**サインイン識別子：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| サインイン識別子 | **Eメール** | ユーザー名よりメールの方がユーザーが覚えやすく、一意性が保証される |

> **「Eメールのエイリアスによるサインインを許可する」は設定しない：** ユーザーが後からメールを変更できてしまう場合に挙動が複雑になるため。初期設計を単純に保つ。

> **🚨 変更不可の設定（重要）：** サインイン識別子（ここで設定するEメールなどの項目）は **ユーザープール作成後に変更できない**。「やっぱり電話番号でもサインインできるようにしたい」と思っても、プールを削除して作り直すしかない。間違えたらユーザープールごと作り直しになるので慎重に選択すること。

**リターンURL（Return URL）：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| リターンURL | （空欄のまま、または `http://localhost:3000/callback`） | ホストされた認証ページ（Managed Login）を使わないため設定不要。空欄で進めてよい |

> **なぜリターンURLが不要か：** TaskFlowでは独自のReact UIで認証画面を作るため、Cognitoのホストされたページにリダイレクトしない。リターンURLはそのリダイレクト先を設定するものなので今回は不要。後から追加・変更も可能。

設定が終わったら **「次へ」（または「概要に移動」）** をクリック。

---

#### Step 1-2: サインイン体験の設定（Configure sign-in experience）

> このステップが表示されない場合や、Step 1-1の中に統合されている場合もあります。画面の内容に応じて進めてください。

**パスワードポリシー：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| 最小の長さ | 8 | 短すぎると総当たり攻撃に弱い。8文字は一般的な最低ライン |
| 大文字・小文字・数字・特殊文字 | 必須のまま（デフォルト） | 適度に複雑にすることで推測しにくくなる。ただし過度な要件はUX悪化につながる |
| 一時パスワードの有効期限 | 7日 | 長すぎると未使用アカウントのリスク。7日は標準的 |

**Multi-factor authentication：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| MFA | **なし** | 学習環境では設定の複雑さを避ける。本番（特に管理者アカウント）ではTOTP（Google Authenticator等）を有効化推奨 |

**「次へ」**

---

#### Step 1-3: サインアップ体験の設定（Configure sign-up experience）

**ユーザーアカウントの復旧：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| セルフサービスのアカウント復旧 | 有効 | ユーザーが自分でパスワードリセットできる機能。なければ全て管理者が対応する必要がある |
| 検証メッセージ配信方法 | Eメールのみ | SMS（電話番号）は別途SNSサービスの設定と費用が発生するため、今回はメールのみ |

**サインアップ設定：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| セルフサービスのサインアップ | 有効 | ユーザーが自分でアカウントを作れる。招待制にしたい場合は無効にして管理者が作成する |
| Eメールで検証 | **チェック** | 本物のメールアドレスを確認することでスパム登録を防ぐ |

**「次へ」**

---

#### Step 1-4: メッセージの配信設定（Configure message delivery）

| 項目 | 値 | 判断理由 |
|------|----|---------|
| メール送信 | **Cognito でメールを送信** | 学習・開発環境向け。1日50件の制限あり。本番ではAmazon SES（送信制限なし）を設定する |

> **本番でSESを使う理由：** Cognitoの組み込みメールは配信制限があり、送信元のドメインがAWSのドメインになる。SESを設定すると自社ドメインから送れる。

**「次へ」**

---

#### Step 1-5: アプリの統合（Integrate your app）

このステップでユーザープール名とアプリクライアントの詳細設定を行います。

**ユーザープールの統合：**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| ユーザープール名 | `taskflow-user-pool` | |

> **「ホストされた認証ページを使用」について（UI変更の注意）：**
>
> 旧UI（2024年10月以前）では、このステップに「ホストされた認証ページを使用」というチェックボックスが表示されていました。
>
> **新UI（2024年11月以降）では、このチェックボックスは表示されません。** 代わりに、Hosted UI（現在は「Managed Login」と呼ばれる）はユーザープール作成後に別途設定する方式に変わりました。
>
> TaskFlowでは独自のReact UIで認証画面を作るため、Hosted UI / Managed Loginは使いません。このステップで何も追加設定せず、そのまま「次へ」で進んで構いません。
>
> もしチェックボックスが表示されている場合は、**チェックしないまま**進んでください。

**アプリクライアントの確認：**

Step 1-1で「Single-page application」を選んだため、以下の設定が自動的に入力されているはずです：

| 項目 | 期待される値 | 確認方法 |
|------|------------|---------|
| アプリクライアント名 | `taskflow-web-client`（Step 1-1で入力した値） | 表示を確認 |
| クライアントシークレット | **生成しない**（なし） | SPAを選んだので自動的にシークレットなし |
| 認証フロー | `ALLOW_USER_SRP_AUTH` が含まれること | SRP = セキュアリモートパスワード。パスワードをそのまま送らず安全に認証できるプロトコル |

> **アプリクライアント名を変更したい場合：** Step 1-1で入力した名前が引き継がれています。このステップで変更も可能です。

**タグ：**（ページ下部のタグセクションに設定）

| キー | 値 |
|------|-----|
| Name | taskflow-user-pool |
| Environment | dev |
| Project | taskflow |
| ManagedBy | manual |

> **ユーザーグループへのタグ設定について：** CognitoのユーザーグループはAWSがタグ付けをサポートしていないため、タグの設定は不要です。

**「次へ」**

**確認すべき主要項目：**

- アプリクライアント: `taskflow-web-client`（クライアントシークレットなし）

### Step 2: ユーザーグループの作成

> 図: Cognitoの認証フロー（ログイン → JWTトークン → API認可）

```mermaid
sequenceDiagram
    participant User as ユーザー (ブラウザ)
    participant React as React App
    participant Cognito as Cognito\nUser Pool
    participant API as ECS Backend\n(Node.js API)

    User->>React: メール + パスワード入力
    React->>Cognito: SRP認証リクエスト\n(パスワードを平文送信しない)
    Cognito-->>React: IDトークン + アクセストークン\n(JWT形式)
    
    Note over React: JWTをローカルストレージに保存
    
    React->>API: APIリクエスト\nAuthorization: Bearer <JWTトークン>
    API->>Cognito: トークン検証\n(公開鍵で署名確認)
    Cognito-->>API: ユーザー情報 + グループ情報\n(cognito:groups: ["Admin"])
    API-->>React: レスポンス\n(グループ権限に応じた結果)

    Note over API: グループで権限判定\nAdmin > User > Guest
```

1. 作成した `taskflow-user-pool` をクリック → **「グループ」タブ** → **「グループを作成」**

以下の3グループを順番に作成：

| グループ名 | 説明 | 優先順位 |
|-----------|------|---------|
| `Guest` | 閲覧のみ | 3 |
| `User` | 通常ユーザー（タスクの作成・編集） | 2 |
| `Admin` | 管理者（全操作） | 1 |

> **IAMロールは設定しない（今回）：** グループにIAMロールを紐づけるとAWSリソースへの操作権限をユーザーに付与できる（Cognito Identity Pool連携時）。今回はアプリ内での権限制御にグループ名を使うだけなのでIAMロールは不要。

> **優先順位の意味：** ユーザーが複数グループに所属している場合、`cognito:preferred_role` の決定に使われる。アプリで最初に評価するグループを優先順位1に設定する。

### Step 3: テストユーザーの作成

1. **「ユーザー」タブ** → **「ユーザーを作成」**

| 項目 | 値 | 判断理由 |
|------|----|---------|
| ユーザーを招待する方法 | **パスワードを送信しない** | テスト用なので招待メールは不要 |
| Eメールアドレス | 任意（自分のアドレス） | |
| 一時パスワード | 自分でパスワードを設定 | |
| Eメールを確認済みとしてマーク | **チェック** | メール確認フローをスキップするため |

2. 作成したユーザーをクリック → **「ユーザーをグループに追加」** → `Admin` を選択

## 確認ポイント

1. ユーザープールの **「ユーザープールID」** を確認・メモ（形式: `ap-northeast-1_Xxxxxxxxx`）
2. **「アプリクライアント」** タブで `taskflow-web-client` の **クライアントID** をメモ
3. アプリクライアントの詳細で **「クライアントシークレット」が「なし」** であることを確認（SPAなのでシークレット不要）
4. **「グループ」** タブに `Guest`・`User`・`Admin` の3グループが作成されているか
5. テストユーザーのステータスが **「確認済み」** になっているか
